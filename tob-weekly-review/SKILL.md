# TOB Weekly Review

## Purpose

把分散在需求、提交、PR、测试反馈和发布记录里的 TOB 研发信息，整理成一份可复盘的周报，并形成风险 action 闭环。

这个 skill 的目标不是泛泛总结提交，而是帮助开发负责人每周回答：

1. 本周业务推进到了哪里。
2. 当前核心后端工作和相关业务模块分别完成了什么。
3. 哪些跨仓库、跨模块、跨角色依赖需要关注。
4. 哪些风险要转成明确 action，并在下周回看。

## Guardrails

- 不要把当前后端工作重点写成 TOB 业务的全部边界。可以写“当前核心后端工作”“相关业务模块”“当前重点仓库”，不要写成“TOB 只包含某几个项目”。
- 不要按仓库机械堆提交。先按业务能力归类，再把仓库、分支、PR、提交作为证据。
- 不要把 AI 推断写成事实。没有证据的内容标为“待确认”。
- 不要只写完成事项。必须写风险、阻塞、下周动作和上周 action 回看。
- 不要编造 KPI、上线效果、事故结论或用户影响。没有明确材料时写“当前材料未体现”。

## Inputs

优先收集这些材料：

- 本周 git log、PR/MR、merge 记录、分支名。
- 本周需求文档、技术方案、测试反馈、发布记录。
- 本周线上问题、联调问题、阻塞点。
- 上周周报中的 action，用于本周回看。
- 下周计划、待发布版本、待评审方案。

如果材料不足，先输出“可用材料”和“缺失材料”，再继续做保守版周报。

## Workflow

### 1. Determine Scope

确认本周范围：

- 周期：例如 `2026-W13` 或 `2026-03-24 至 2026-03-28`
- 业务范围：例如辅学教练工作台、掌握度/PT-Z、学习报告、新机批、错题本
- 技术范围：当前核心后端工作、相关前端/客户端、数据服务、测试发布
- 产物路径：默认 `doc/weekly/tob/YYYY-WW.md`

### 2. Gather Evidence

按来源提取证据，不要直接下结论：

- 需求证据：需求名、文档路径、关键目标、边界。
- 代码证据：仓库、分支、PR/MR、提交、关键类/模块。
- 测试证据：测试环境、回归结果、阻塞问题。
- 发布证据：发布分支、上线窗口、回滚风险。

推荐用表格记录：

```markdown
| 来源 | 证据 | 说明 |
|---|---|---|
| git | <repo>/<branch>/<commit> | <业务含义> |
| PR | #123 | <合入范围> |
| doc | <path> | <需求边界> |
```

### 3. Group By Business Capability

先按业务能力分组，再归纳仓库变更：

- 辅学教练工作台
- 虚拟教室 / 签到 / 公屏
- 实时监测 / 事件曲线 / 学生四分类
- 学习链路 / 掌握度 / PT-Z
- 错题本 / 学习报告 / 新机批
- 工程治理 / 服务隔离 / 发布联调

每组写清：

- 本周完成什么。
- 当前状态是什么。
- 依赖谁。
- 下周要推进什么。

### 4. Identify Risks

风险必须具体到机制，不要只写“有风险”：

- 跨仓库依赖未合齐。
- 接口契约或 DTO 字段未对齐。
- MQ / WebSocket / MQTT 边界不清。
- 幂等、重复投递、重试、补偿未验证。
- 统计口径或数据来源未确认。
- 大提交、模糊 commit message、缺少测试导致回溯困难。
- 联调、测试、发布、回滚路径不清。

每个风险转成 action：

```markdown
| 风险 | 影响 | Owner | Action | 截止时间 | 状态 |
|---|---|---|---|---|---|
| <风险机制> | <影响范围> | <负责人> | <下一步动作> | <日期> | Open |
```

### 5. Review Last Week Actions

如果存在上周周报，必须回看上周 action：

- Done：已经关闭，写证据。
- Carry Over：未关闭，说明原因和新的截止时间。
- Dropped：确认不再做，说明原因。

### 6. Produce Weekly Markdown

默认写入 `doc/weekly/tob/YYYY-WW.md`。如果用户没有授权写文件，先在对话中输出草稿。

输出模板见 `references/weekly-template.md`。

可以用脚本生成空白骨架：

```bash
python scripts/new_weekly_review.py --week 2026-W13 --out doc/weekly/tob/2026-W13.md
```

### 7. Final Check

交付前检查：

- 周报是否按业务能力组织，而不是只按仓库列提交。
- 每个关键结论是否有材料或代码证据。
- 是否避免把当前重点仓库写成 TOB 全部边界。
- 是否包含风险表和 action 表。
- 是否包含上周 action 回看。
- 是否有下周动作。

## Output Contract

最终输出一份 Markdown 周报，至少包含：

1. 本周核心结论
2. 本周业务进展
3. 当前核心后端工作进展
4. 跨模块 / 跨仓库依赖
5. 风险与阻塞
6. 测试与发布关注点
7. 上周 action 回看
8. 下周动作
9. 需要沉淀的文档 / 方案 / 复盘
10. 证据索引

如果是第一次落地这个流程，可以先产出第 0 期试运行周报，并在结尾标记“流程试运行”。
