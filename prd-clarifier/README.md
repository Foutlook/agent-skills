# prd-clarifier

把 PRD、PDF、网页说明、截图、流程图、时序图、表格等混合需求材料，梳理成一份更明确、更严谨、可被 `Codex`、`Claude` 继续消费的 Markdown 规范文档。

这个 skill 不只是把问题列出来。它会先产出结构化草稿，再把高影响的未决问题按优先级逐条向用户确认，并把确认结果回写到主文档。无法确认的内容会明确标成 `未确认` 或留空，不会伪装成已确认需求。

如果 PRD 里的图片、截图或外链图片出现，它默认会先尝试下载并打开查看；如果下载、打开或读取失败，也不会中断，而是会：

- 主动记录图片来源位置、访问方式、是否已下载、是否已打开查看
- 遇到权限限制时主动申请权限，而不是直接跳过图片
- 在主文档里记录图片链接、拉取状态、失败原因和缺失影响
- 在图片章节中显式记录权限状态、拉取状态、打开查看状态
- 把从图片中识别出的关键信息回写到功能需求、页面与交互、关键业务规则等受影响章节
- 主动向用户发起补充提醒
- 优先使用固定的中文追问模板，避免每次随机发挥
- 权限受限时优先使用固定的中文权限申请模板，避免模糊表达
- 提供明确补充方式：重新上传、文字描述或新链接
- 跟踪补充状态：`待补充` / `已补充待回写` / `已回写`

## 适用场景

- 产品给出的是 PRD、飞书文档、网页、PDF、截图混合材料
- 产品给出的是一句话需求或产品想法，需要先引导再澄清
- 你希望先把需求梳理清楚，再交给 AI 继续生成计划、拆任务、写代码、补测试
- 你希望模型帮助暴露逻辑冲突、边界缺失和模糊规则
- 你不希望模型擅自脑补缺失需求，而是显式列出歧义、推导结论和待确认项
- 你希望模型对关键歧义逐条发问并在确认后更新文档
- 你希望同一模块、同优先级、低耦合的问题可以批量确认以提高效率
- 你希望模型自动检测权限、状态、时间范围、入口、验收之间的明显冲突
- 你希望确认循环有明确的终止条件，不会无限追问

## 如何使用

### 1. 在支持 skills 的环境中安装

如果你要给 `Codex` 用，通常放到个人 skills 目录，例如：

```powershell
Copy-Item -Recurse .\skills\prd-clarifier $env:USERPROFILE\.codex\skills\
```

如果你要给 `Claude Code` 用，通常放到个人 skills 目录，例如：

```powershell
Copy-Item -Recurse .\skills\prd-clarifier $env:USERPROFILE\.claude\skills\
```

如果你的环境有自定义 skill 搜索路径，就复制到对应目录即可。

```text
把这份 PRD 梳理成 AI 能理解的 Markdown 规范文档，帮我把需求结构理顺、逻辑补齐、冲突暴露出来。遇到关键歧义时请逐条问我确认，并把确认结果回写到文档里，后续要给 Codex 继续拆任务和写代码。
```

```text
我会给你一个 PDF 摘要、一张后台截图和两个链接。请按固定模板整理，不只是转写，还要把需求讲清楚。图片请自动下载并打开查看；如果没有权限就主动申请。待确认问题请按优先级一步步问我，确认后的结论直接更新文档。
```

```text
我现在只想快速看这份需求的 6 条摘要，不需要完整模板，也不要继续追问确认。
```

```text
如果同一模块里有 2-3 个低耦合待确认点，可以合并成一轮批量确认；但权限、状态流转和关键冲突还是优先单条确认。
```

```text
我只有一句话需求"我想做个签到功能"，帮我整理成后续能用的文档。
```

```text
这份需求的确认问题我暂时回答不了，请按终止规则输出当前文档状态。
```

### 3. 输出预期

输出是单一主 Markdown 文档，重点不是“照抄 PRD”，而是“把需求梳理清楚后再输出”。工作方式通常分四段：

1. 先生成一版结构化草稿，显式列出缺失、冲突和待确认项。
2. 先处理图片、截图和图示，能下载的先下载并打开查看；遇到权限限制先申请，失败时记录状态、原因和影响。
3. 自动检测明显冲突，特别是权限、状态、时间范围、入口、验收之间的矛盾。
4. 对高影响问题逐条确认；对同模块、同优先级、低耦合问题可做一轮批量确认。
5. 把确认结果和图片识别出的关键信息回写到主文档对应章节，无法确认的保留为 `未确认`。

在“完整澄清模式”下，内容包含：

- 文档信息
- 需求目标与原始内容摘要
- 术语与对象
- 功能需求
- 关键业务规则与约束
- 非功能与约束
- 页面与交互
- 流程与时序
- 表格转录
- 图片与图示说明
- 链接与引用资料
- 验收标准
- 未决问题与歧义
- AI 执行提示
- 下游 Skill 数据契约

如果用户明确只要轻量摘要，或明确要求直接进入实现/任务拆分，则不应强行输出完整 15 章节文档，而应尊重用户目标，仅保留必要风险提示。

### 4. 图片权限申请模板

当图片不是失效，而是需要额外权限时，优先使用这段固定模板：

```text
图片“<图片标题>”当前不是链接失效，而是需要额外权限才能下载或打开。我需要先拿到访问权限，才能继续确认 <受影响内容>。

如果你同意，请授权我继续拉取这张图片；如果暂时不能授权，我会先把相关内容保留为 `未确认`，并在文档中记录权限阻塞和影响范围。
```

## 设计原则

- 先保证业务含义不跑偏，再做需求澄清与严谨化整理
- 多模态信息统一收敛到一个 Markdown 主文档
- 图片默认主动下载并打开查看，不能停留在“有图未读”
- 允许对散乱内容做归并、拆分和结构重组
- 对缺失、冲突、歧义零脑补
- Mermaid 仅在可可靠还原时使用；否则退化成结构化文字说明
- 文档结构固定，方便下游 AI 连续消费
- 高影响歧义必须转成可回答的问题，而不是停留在问题清单
- 同优先级、同模块、低耦合问题可批量确认，但每批建议不超过 3 个
- 自动检测明显冲突，并把冲突来源和影响显式写出来
- 用户一旦确认，相关章节必须同步回写
- 图片中识别出的关键事实必须同步回写到主章节，不能只留在图片说明区
- `功能需求` 中要显式留出 `推断` 和 `冲突信息`
- `待逐条确认问题` 中要显式写出“为什么需要确认”
- 模糊输入只做 1 轮引导 + 1 轮聚焦追问，之后必须收敛成“待补充需求骨架”或终止
- 用户授权“你决定/按常规处理”时，只能落在 `P1/P2` 的暂行写法，不能替代 `P0` 产品决策

## 评测

基础评测样例见 [evals/evals.json](C:/idea_workspace_tob/skills/prd-clarifier/evals/evals.json)。

当前评测重点：

- 能否保持固定模板并与第 15 章下游契约一致
- 能否让需求表达更明确、逻辑更严谨
- 能否保留链接、图表、表格和图片信息
- 能否默认下载并打开图片；遇到权限限制时主动申请
- 能否把图片中的关键事实回写到主章节，而不是只写在图片说明里
- 能否自动检测明显冲突并记录来源
- 能否把缺失、冲突、待确认项显式列出
- 能否把高影响待确认项转成逐条确认问题
- 能否在同模块低耦合场景下合理使用批量确认
- 能否在无法确认时保留 `未确认`，避免静默补全
- 能否在模糊输入时先引导再澄清
- 能否在确认循环超出限制时正确终止
- 能否正确处理用户授权推断场景，且不替代 `P0` 决策
