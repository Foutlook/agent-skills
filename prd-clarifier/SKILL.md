# PRD Clarifier

## Overview

把混合格式需求材料整理成一份可继续交给 AI 执行的 Markdown 规范文档。

这个 skill 的目标不是“转写 PRD”，而是“保真前提下做结构化澄清”。它必须显式区分原始事实、整理后归纳、推断、冲突信息和未确认项，不能静默脑补。

## When to Use

适用：

- 用户给的是 PRD、PDF、网页、截图、流程图、时序图、表格或混合材料
- 材料存在重复、散乱、冲突、字段缺失、规则不清、边界不明
- 用户希望先把需求梳理清楚，再交给 AI 做 implementation plan、任务拆分、编码或测试
- 用户希望模型主动暴露冲突、缺失和高影响待确认项，并形成确认闭环

不适用：

- 用户要你从零发明产品方案
- 用户明确只要轻量摘要
- 用户给的需求已经足够清晰，且明确要求直接进入实现或任务拆分

## Core Rules

- 保留原始业务意图，但不要保留原始混乱表达。
- 可以重组、拆分、合并描述，但不能把缺失规则写成已确认事实。
- 对跨来源整理出的结论标记 `整理后归纳`；超出原文直接表述的内容标记 `推断`。
- 发现互相矛盾的信息时，显式标记 `冲突信息`，并注明来源。
- 高影响未决项必须转成可直接回答的确认问题，不能只给模糊问题清单。
- 用户每回答一次，都要把受影响章节立即回写，不能只在问题列表追加日志。
- 图片、截图、表格、链接、流程图、时序图都属于需求材料，不能因为难处理就丢掉。
- 文档中的图片默认要主动下载并打开查看，不能停留在“有图但未读”状态，也不能等用户催促。
- 下载、打开或读取图片遇到权限限制时，要主动申请权限；如果最终仍无法处理，必须记录状态、原因、影响和补充动作。
- 从图片中读到的关键事实，必须回写到对应业务章节和 `图片与图示说明`，不能只写“见图”或只停留在分析过程里。
- 只有在流程或时序可可靠还原时才输出 Mermaid；如果不能可靠还原，禁止输出占位 Mermaid，改为结构化文字并明确缺失点。
- 输入很弱时也要保留完整结构，但用 `原始材料未明确`、`未确认`、`内容未实际获取` 等占位，不得编造。

## Workflow

### 1. 盘点输入

- 列出每个来源及类型：文本、PDF、网页、图片、表格、流程图、时序图、链接
- 先说明已拿到什么、缺什么、哪些内容不可读
- 对文档中的图片，额外标记其来源位置、访问方式、是否已下载、是否已打开查看
- 能直接拉取的图片先拉取再继续整理；需要权限的图片先申请权限，再决定后续补充路径

#### 1.1 输入强度判断

根据输入完整度，选择不同处理模式：

| 输入强度 | 特征 | 处理模式 |
|---------|------|---------|
| **完整** | 有明确功能描述、角色、页面、验收标准 | 标准澄清模式 |
| **部分** | 有功能描述但缺角色/权限/边界 | 补充引导模式 |
| **模糊** | 只有一句话需求或产品想法 | 先引导后澄清 |

#### 1.2 模糊输入引导

如果输入不是正式 PRD 而是以下类型，必须先引导用户补充核心信息：

| 输入类型 | 示例 | 引导重点 |
|---------|------|---------|
| 口头描述 | "我想做个签到功能" | 目标用户、核心页面、触发条件 |
| 产品想法 | "这个功能应该很有趣" | 解决什么问题、目标用户、预期价值 |
| 竞品截图 | "我们要做成这样" | 提取可见元素、询问差异点、确认核心流程 |
| 单句需求 | "支持导出学习报告" | 导出内容、触发条件、权限角色、格式限制 |

**模糊输入引导模板：**

```text
你提供的是产品想法而非正式 PRD。我先确认几个核心点，确认后我会生成一份需求骨架供你补充：

1. 这个功能解决什么问题？
2. 目标用户是谁？
3. 核心页面或流程是什么？
4. 有没有类似产品或功能可以参考？

请按序号回复，或直接补充你已知的信息。
```

引导完成后，基于用户回答生成需求骨架，再进入标准澄清流程。

#### 1.3 模糊输入收敛规则

模糊输入不能无限引导，必须收敛：

- 第一轮只允许使用 1 组引导问题，最多 4 个问题
- 如果用户回答后仍然缺少核心信息，只允许再补 1 个聚焦追问
- 如果补 1 轮后仍无法形成稳定的角色、目标或核心流程，就停止继续追问
- 停止后输出一份“待补充需求骨架”，把已知信息写入主文档，把空白点保留在 `未决问题与歧义`
- 除非用户明确要求继续头脑风暴，否则不要把模糊输入阶段拖成开放式产品设计讨论

### 2. 先搭骨架

- 主文档必须使用 `references/output-template.md` 的固定章节顺序
- 先建完整结构，再逐节回填，避免图片、表格、约束或验收信息被漏掉

### 3. 抽取并澄清需求

每个功能点至少检查这些信息：

- 原始描述
- 整理后归纳
- 推断
- 冲突信息
- 前置条件
- 用户操作
- 系统行为
- 输出结果
- 决策条件
- 异常或边界处理
- 业务约束
- 来源引用

如果某项原文没有提供，不要凭经验补齐，明确写 `原始材料未明确` 或 `未确认`。

### 4. 统一处理非文本内容

图片或截图：

- 不能只写“见图”
- 文档中的远程图片、嵌入图片、附件图片，默认都要主动下载到本地并打开查看；多张图片逐张处理
- 下载或打开前如果权限不足，要立即申请，不要因为当前无权限就直接跳过
- 提取页面区域、字段、按钮、标签、状态差异、空态、错误提示、标注
- 把图片中可确认的关键信息回写到对应章节，例如 `功能需求`、`页面与交互`、`关键业务规则与约束`，并在 `图片与图示说明` 中保留来源和摘要
- 无法读取时，保留原图来源，记录拉取状态、失败原因、缺失影响和补充状态
- 每次只追一个未解决的图片补充问题

图片补充追问优先使用这段模板：

```text
图片“<图片标题>”目前无法读取，会影响我确认 <受影响内容>。

请你任选一种方式补充：
1. 重新上传这张图片
2. 直接用文字描述图片里的字段、按钮、状态、提示文案和标注
3. 提供一个新的可访问图片链接
4. 先保留为未确认，后续再补
```

图片权限申请优先使用这段模板：

```text
图片“<图片标题>”当前不是链接失效，而是需要额外权限才能下载或打开。我需要先拿到访问权限，才能继续确认 <受影响内容>。

如果你同意，请授权我继续拉取这张图片；如果暂时不能授权，我会先把相关内容保留为 `未确认`，并在文档中记录权限阻塞和影响范围。
```

表格：

- 转成 Markdown 表格
- 列含义不清时保留原表头，并把歧义写入 `未决问题与歧义`

流程图与时序图：

- 可可靠还原时转成 Mermaid
- 不可可靠还原时，改写为步骤、决策点、参与方、输入输出、缺失条件

链接：

- 保留原 URL
- 标注用途，例如原始 PRD、原型、接口说明、补充业务说明

### 5. 自动冲突扫描

至少检查这些冲突：

- 角色或权限冲突
- 页面入口、可见性或渠道冲突
- 状态与操作冲突
- 时间、次数、范围、阈值冲突
- 字段定义或表格列不一致
- 原型、截图和正文描述不一致
- 验收条件与行为描述不一致
- 默认规则与例外规则不一致

高风险冲突需要升级成确认问题，不能先替用户选边站。

### 6. 未决项分级

- `P0`：阻塞架构、编码、数据模型、权限、验收逻辑
- `P1`：影响行为分支、异常处理、边界规则、范围判断
- `P2`：影响文案质量、次要交互或非阻塞元信息

### 7. 确认闭环

- 默认从最高影响未决项开始确认
- `P0`、权限冲突、状态流转冲突、强耦合决策，默认一次问一个
- 只有同模块、同优先级、低耦合、可独立回答的问题，才允许 2-3 个合并一轮批量确认
- 每个确认问题都要写清：问题本身、为什么需要确认、影响章节、候选解释
- 用户部分回答时，只回写已确认项，未回答部分继续保留
- 用户表示不知道时，写 `未确认`，不要代答

批量确认可用这个简版模板：

```text
下面是同一模块的 2-3 个待确认点，我合并一轮确认以提高效率。请按序号直接回复即可；如果某一项现在无法确认，请回复”未确认”。

1. <问题 1>
2. <问题 2>
3. <问题 3>
```

#### 7.1 确认循环终止规则

确认循环不能无限进行，必须遵循以下终止规则：

**按优先级设置追问上限：**

| 优先级 | 最多追问次数 | 超限处理 |
|-------|-------------|---------|
| P0 | 2 次 | 标记 `未确认`，保留暂行写法，文档状态改为 `部分已确认` |
| P1 | 1 次 | 标记 `未确认`，继续处理下一项 |
| P2 | 1 次 | 标记 `未确认`，可跳过 |

**用户授权推断：**

当用户回复以下内容时，可在有限范围内形成暂行写法：
- “你决定”
- “按常规处理”
- “看着办”
- “我不确定，你定吧”

**用户授权推断边界：**
1. 只允许用于 `P1/P2` 的次级默认项、文案口径、非核心交互细节或结构化占位
2. 禁止用于 `P0`、角色权限、状态流转、验收标准、计费/合规/安全规则、核心数据模型、外部接口契约
3. 即使用户授权，也只能写成 `暂行写法（用户授权）`，不能提升为已确认需求
4. 如果用户把 `P0` 决策丢给模型，只能记录为 `待产品拍板`，并可附 `建议方案`，不能直接定稿

**推断记录规则：**
1. 必须基于行业通用做法或同类产品惯例
2. 必须在问题记录中标记 `暂行写法（用户授权）`
3. 必须在 `为什么需要确认` 字段注明“用户授权推断，仍待最终确认”
4. 必须在 `当前暂行写法` 字段说明推断依据和适用边界

**强制终止条件：**

出现以下任一情况，必须终止确认循环并输出当前文档：
1. 单个 P0 问题累计 2 次追问未果
2. 用户明确说”停止追问”或”先这样”
3. 用户连续 3 次回复”未确认”或”不知道”
4. 确认轮数超过 5 轮
5. 模糊输入经过 1 轮引导 + 1 轮聚焦追问后，仍无法形成稳定需求骨架

**终止后的文档状态：**
- 确认状态改为 `部分已确认` 或 `确认未完成`
- 在 `AI 执行提示` 中明确标注未完成的确认项
- 下游 Skill 应将未确认项视为风险点而非阻塞点
- 对模糊输入场景，明确标注“当前仅为待补充需求骨架”

#### 7.2 确认问题质量标准

每个确认问题必须满足：
- 问题本身不超过 2 行
- 候选选项使用 A/B/C/D 格式，每个选项不超过 1 行
- 必须有 `为什么需要确认` 字段
- 必须有 `影响章节` 字段
- 批量确认必须明确”可部分回答”

## Output Contract

- 主文档章节顺序必须与 `references/output-template.md` 完全一致
- 章节可以留空，但空位必须写成 `原始材料未明确`、`未确认`、`内容未实际获取` 或等价的保守表述
- `功能需求` 中要给 `推断` 和 `冲突信息` 留稳定位置，不能只在脑内区分
- 图片读取得到的关键信息，不能只放在 `图片与图示说明`，还必须回写到受影响的主章节
- `待逐条确认问题` 中要包含“为什么需要确认”或等价字段，不能只写问题本身
- `流程与时序` 章节禁止输出为了凑模板而编出来的 Mermaid

## Final Check

返回前自检：

- 没有忽略任何主要来源类型
- 没有把缺失信息写成已确认事实
- 冲突信息有来源，有影响，有后续处理方式
- 高影响未决项已被提问，或明确保留未确认
- 已确认答案已回写到对应章节
- 文档中的图片已下载并打开查看，或已明确记录权限申请/失败原因/缺失影响
- 从图片中提取的关键事实已写入主文档，而不是只留下图片链接或“见图”
- 图片拉取失败记录了状态、原因、影响和补充动作
- 输出仍然是单一主 Markdown 文档

## Common Mistakes

- 把需求澄清文档写成新的产品方案
- 只做语言润色，没有把逻辑结构和边界理清
- 只列问题，不发起确认
- 用户答复后只更新问题区，不回写主文档
- 用占位 Mermaid 假装流程已澄清
- 因为图片或表格难处理就省略
