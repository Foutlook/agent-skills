## User Input

```text
$ARGUMENTS
```

先使用 `prd-clarifier` skill。如果该 skill 不可用，按本命令中的规则执行。

目标：

- 把当前对话中的混合需求材料整理成一份单一主 Markdown 文档
- 显式区分原始事实、整理后归纳、推断、冲突信息和未确认项
- 让后续 `Codex`、`Claude` 可以继续做计划、拆任务、编码和测试

必须遵守：

1. 不擅自补全业务规则、接口字段、权限结论或验收标准。
2. 文字、图片、截图、链接、表格、流程图、时序图都要保留和处理。
3. 文档中的图片默认要主动下载到本地并打开查看；如果下载、打开或读取需要权限，要主动申请权限，不能直接跳过。
4. 流程或时序只有在可可靠还原时才输出 Mermaid；否则改为结构化文字说明，不允许输出占位 Mermaid。
5. 缺失、冲突、不可读、未确认的信息，统一进入 `未决问题与歧义`，并在受影响章节保守落位。
6. 不允许只写“见图”或“见链接”，必须提取可见事实或说明用途；图片中的关键信息还必须回写到受影响主章节。
7. 如果用户明确只要轻量摘要，或明确要求直接进入实现/任务拆分，则不要强行套完整澄清模板。
8. 如果走完整澄清文档，章节顺序必须与 `references/output-template.md` 一致。
9. 如果输入是模糊想法而不是正式 PRD，先做一轮最多 4 个问题的引导；若补 1 轮后仍然模糊，就输出“待补充需求骨架”，不要无限追问。
10. 用户授权“你决定/按常规处理”时，只能对 `P1/P2` 非核心细节形成 `暂行写法（用户授权）`；禁止用来拍板 `P0` 权限、状态流转、验收、安全或接口契约。
11. 如果确认循环达到终止条件，输出当前文档并把状态标为 `部分已确认` 或 `确认未完成`。

如果用户输入里带了额外要求，把它视为本次转换的补充约束，但不要覆盖以上保真规则。

完整文档模式输出前自检：

- 是否遗漏了任一输入材料
- 是否把猜测写成了事实
- 图片是否已下载并打开查看，或已明确记录权限申请/失败原因/缺失影响
- 是否在功能需求中给出了 `推断` 和 `冲突信息`
- 图片中的关键事实是否已回写到主章节，而不只停留在“图片与图示说明”
- 是否在待确认问题中写清了“为什么需要确认”
- 是否误把 `P0` 的用户授权当成已确认需求
- 模糊输入是否已经在 1 轮引导 + 1 轮追问后收敛
- 是否把无法可靠还原的流程/时序错误地写成了 Mermaid
- 是否保持了固定模板
