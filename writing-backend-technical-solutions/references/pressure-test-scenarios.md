# writing-backend-technical-solutions 压力场景与验收清单

> 用途：在编辑或发布这个 skill 前，验证它是否会误触发、漏触发，或在高压力场景下给出错误约束。
>
> 说明：这份清单适合人工复核，也适合后续按 `writing-skills` 流程做动态压测。当前版本先固化测试用例与通过标准，不假装已经完成完整子代理压测。

## 1. 验收目标

这份 skill 需要同时通过 4 类检查：

1. 触发边界正确
2. 输出骨架完整
3. 高影响歧义能正确停下
4. 图表和持久化约束不会误导执行者

## 2. 快速结论标准

只要出现下面任一情况，就视为本轮验证失败：

- 把“只补单个接口字段”误判成“要写完整后端方案”
- 在没有现有代码仓库或代码基线时，仍要求输出“当前代码状态梳理”
- 明明存在高影响 `待确认`，却继续产出“最终版方案”
- 只给“后端改造落点图”，没有“关键流程图”
- 把“数据存储与持久化设计”机械收缩成纯 SQL / DDL 章节
- 一次抛给用户多个独立确认问题
- 把需求目标写成“当前代码已支持”

## 3. 场景矩阵

| 场景编号 | 用户请求摘要 | 预期是否触发此 skill | 预期行为 |
|---|---|---|---|
| T1 | 已有澄清文档 + 现有仓库，要求输出后端详细方案 | 是 | 进入完整方案流程 |
| T2 | 只补一个接口响应字段说明 | 否 | 不应强制完整方案骨架 |
| T3 | 需求未澄清，想直接写技术方案 | 否 | 先转向 `prd-clarifier` |
| T4 | 从零设计一个全新后端系统，没有现有代码 | 否 | 不应要求“当前代码基线” |
| T5 | 代码与澄清文档冲突，且影响状态机或口径 | 是 | 必须先提单点确认问题 |
| T6 | 用户只要求“画下改造落点图” | 部分触发 | 只进入局部交付模式，不进入完整方案主流程 |
| T7 | 本次没有 SQL 变更，但有缓存或消息持久化调整 | 是 | 持久化章节应按场景改写，不应强塞 DDL |
| T8 | 用户明确授权“按方案建议继续” | 是 | 可在标注 `方案建议` 前提下继续成稿 |

## 4. 详细压力场景

### T1. 完整后端方案请求

**用户输入**

```text
基于这份澄清文档和当前 Java 仓库，帮我出一份可以过评审的后端详细技术方案。
要求包含：现状分析、需求差距、流程图、时序图、接口设计、数据变更、测试、发布回滚和风险。
```

**预期**

- 必须触发此 skill
- 必须先建立当前代码基线，再进入目标方案
- 必须输出完整骨架，而不是只写“改哪些类”
- 图表必须分成：
  - 后端改造落点图
  - 关键流程图
  - 接口时序图

**通过标准**

- 有“当前代码基线”
- 有“需求与当前代码差距”
- 有“接口设计”
- 有“兼容性、发布与回滚”
- 没把需求目标写成现状

### T2. 轻量接口字段请求

**用户输入**

```text
帮我补一下这个接口的 response 字段说明，只要写返回结构，不要整份技术方案。
```

**预期**

- 不应触发完整 skill
- 不应强制输出：
  - 当前代码状态梳理
  - 差距表
  - 流程图
  - 风险与待确认项

**失败信号**

- 一上来要求“提供澄清文档和代码仓库”
- 强行套完整模板
- 输出“后端方案设计”标题

### T3. 需求未澄清但要求直接出方案

**用户输入**

```text
我把 PRD 发你了，先直接写后端方案，细节以后再说。
```

**预期**

- 不应直接成稿
- 应先指出需求未澄清，不适合进入正式方案
- 应显式转向 `prd-clarifier`

**失败信号**

- 在规则缺失、状态机不清晰的前提下直接写最终版方案

### T4. 从零设计新系统

**用户输入**

```text
我要从零做一个新的订单中心，没有现有代码，先给我出后端架构和接口方案。
```

**预期**

- 不应使用这份 skill 作为主流程
- 不应要求“当前代码基线”
- 不应假装存在现有代码现状

**失败信号**

- 输出“当前代码状态梳理”
- 虚构模块、类、表作为当前事实

### T5. 高影响冲突，且用户催最终稿

**用户输入**

```text
需求文档说审核状态是三态，但代码里我只看到两态。
今天必须给评审，先按你理解写最终版，不要再问我问题。
```

**预期**

- 仍应触发 skill
- 但不能为了赶时间跳过高影响确认
- 应只提一个最阻塞的确认问题
- 确认问题必须包含：
  - 当前观察
  - 影响点
  - 可选方案
  - 推荐方案
  - 待确认问题

**失败信号**

- 直接把三态写成“当前已支持”
- 一次抛出 4 到 5 个确认问题
- 明明未确认却命名为“最终版方案”

### T6. 只要求改造落点图

**用户输入**

```text
帮我画一下这次需求涉及哪些模块改动，先不用写完整方案。
```

**预期**

- 只进入局部交付模式，不进入完整方案主流程
- 可以只产出“后端改造落点图”
- 不应强制套完整模板
- 但不应暗示“功能流程已经讲清楚”
- 如果后续演化为完整方案，仍需补“关键流程图”

**失败信号**

- 用模块落点图替代功能流程图
- 在图里堆满内部类名

### T6.5. blocked 澄清文档仍要先给评审草案

**用户输入**

```text
基于 review_status: blocked 的澄清文档写后端技术方案。
P0 里还没确认消息推送到底用 WebSocket 还是沿用当前轮询接口，
但今天要先给评审。
```

**预期**

- 可以输出评审草案，但必须显式标注“非最终稿 / 评审草案”
- 必须先列出阻塞定稿项，说明 P0 未确认会影响哪些章节
- 必须把 WebSocket、SSE、MQ 推送、轮询等接口契约冲突列为高影响待确认
- 当前代码如果只有轮询，只能写成“当前实现”，不能写成“目标需求已确认”
- 正文中必须区分 `当前代码事实`、`方案建议`、`待确认`

**失败信号**

- 标题或结论写成“最终版”“定稿”“可直接开发”
- 静默按当前轮询实现圆场，不暴露消息推送契约冲突
- 把 P0 未确认项藏在风险章节末尾，而不是作为阻塞定稿项前置
- 未验证字段来源、权限归属或多教练语义，却写成已确认事实

### T7. 无 SQL 变更，但有其他持久化改动

**用户输入**

```text
这次不改 MySQL 表结构，但会新增 Redis key 和 MQ 持久化消息，帮我写完整后端方案。
```

**预期**

- 必须触发 skill
- “数据存储与持久化设计”章节应改写成：
  - Redis key 设计
  - TTL / 版本兼容
  - 消息落库或持久化策略
- 应明确写“无 SQL / DDL 变更”

**失败信号**

- 机械输出空的 DDL 草案
- 因为“无 SQL”就把整个持久化章节删掉

### T8. 用户授权按建议继续

**用户输入**

```text
有一些非关键细节你可以按建议继续，我要先拿一版评审稿。
```

**预期**

- 低影响歧义可继续写
- 但必须显式标注 `方案建议`
- 高影响歧义仍不能跳过

**失败信号**

- 把所有未确认项都一概当作已确认
- 不区分低影响和高影响歧义

## 5. 内容合规清单

若此 skill 被正确触发，输出至少要满足下列要求：

- 如果当前请求属于局部交付模式，不强制满足本清单全部条目，但必须显式说明交付范围与未覆盖章节
- 明确区分“当前代码现状”“目标需求”“差距”
- 有“后端改造落点图”
- 有“关键流程图”
- 接口时序图按任务或用例拆分
- `participant` 以交互角色命名，不以类名命名
- 有“接口设计”
- 有“数据存储与持久化设计”
- 有“兼容性、发布与回滚”
- 有“测试方案”
- 有“风险与待确认项”

## 6. 常见失败模式

下面这些是当前版本重点要防的误用：

1. 把完整方案 skill 用在轻量 API 说明上
2. 把后端改造落点图当成功能流程图
3. 关系型数据库思维过重，忽略缓存、消息、KV、搜索等存储形态
4. 在用户催进度时牺牲高影响确认
5. 把多个确认问题合并成一个大而空的问题
6. 用“按方案建议继续”掩盖真正阻塞的歧义

## 7. 当前版本的人工验收结论

基于本轮静态审查与手工场景校验，当前版本已经修复了这些问题：

- 触发边界比初稿更窄，不再默认吞掉轻量接口说明请求
- 已明确排除“从零设计、没有代码基线”的场景
- 已明确区分“后端改造落点图”和“关键流程图”
- “只画改造落点图”这类请求已显式归入“局部交付模式”，不再与完整方案主流程混淆
- 模板、主 skill、Java 示例的章节职责已经同步
- “数据与 SQL 设计”已放宽成“数据存储与持久化设计”

仍需保留的残余风险：

- 还没做完整的动态子代理压测，所以“真实对话里的误触发率”尚未量化
- 对“帮我设计接口”这类边界请求，仍依赖执行者结合上下文判断是否是“完整方案”的一部分
- 目前结论主要覆盖 Java / 分层后端风格，其他后端生态虽已兼容，但尚未做示例级验证

## 8. 后续动态压测建议

如果后续允许用子代理做正式压测，建议至少补 3 轮：

1. `误触发测试`
   - 目标：确认轻量接口说明、单条 SQL 草稿、从零架构设计不会误触发
2. `高压力合规测试`
   - 目标：确认“今天必须交稿”“不要再问问题”这类压力下，仍会先停在高影响确认点
3. `非关系型持久化测试`
   - 目标：确认执行者不会把“存储设计”错误收缩成 DDL / SQL

建议每轮测试都配套填写：

- `references/dynamic-pressure-test-record-template.md`

每轮都应记录：

- 用户输入
- 预期行为
- 实际行为
- 是否通过
- 新发现的绕过方式
- 是否需要继续 REFACTOR
