# Verify Implementation With Test Cases

## Purpose

在方案实现完成后，用测试提供的用例和反馈做一次闭环核对：需求是否被正确理解，方案是否覆盖需求，实现是否按方案落地，测试用例是否与需求口径一致。

这个 skill 的目标不是机械跑测试，也不是看到失败就补兼容逻辑，而是建立“需求 → 方案 → 实现 → 测试用例 → 实际结果”的证据链，判断问题究竟出在需求缺失、方案遗漏、实现偏差、测试用例偏差，还是测试环境/数据问题。

## When to Use

适用场景：

- 用户说“方案已经实现了，按测试用例帮我验证一下”
- 测试同学提供了测试用例、验收用例、缺陷单、回归结果或测试截图，需要判断实现是否满足需求
- 需要核对需求、技术方案、代码改动、测试用例之间是否一致
- 测试失败后，需要判断是代码 bug、需求口径不清、方案没覆盖、测试用例理解错误，还是环境/数据问题
- 发布前需要形成一份可评审的验收对齐报告

不适用场景：

- 需求还没有澄清，连目标业务规则都不稳定；先用 `prd-clarifier`
- 还没有实现代码，只是在写技术方案；先用 `writing-backend-technical-solutions`
- 用户只要求重构代码结构且不涉及验收用例；优先用 `refactor-module-safely`
- 用户只要简单运行现有测试命令，不需要做需求/方案/实现对齐分析

## Core Rules

- 先找直接失败点，再追真实调用链，不要只凭 guard clause、字段名或测试断言下结论。
- 测试用例是验收证据，不自动等同于需求真相。若测试用例和已澄清需求冲突，标记为“测试用例口径待确认”，不要静默改代码迎合。
- 不要用新增 fallback、替代字段、平行查询或兼容分支掩盖源头不一致。除非代码、schema、查询或接口契约证明该 fallback 是真实业务要求。
- 对列表、聚合、统计、图表、映射补全类问题，优先怀疑实体集来源不一致；必须核对响应字段最终赋值点、实体集来源和 mapping 来源。
- 涉及批量数据时禁止提出 N+1 修复方案。先收集批量参数，再批量查询或批量调用。
- 结论必须包含证据链：失败点、需求依据、方案依据、实现路径、最终数据源/计算点、关键参数、测试证据、分类结论。
- 如果需要改代码，选择能闭环修正错误限制或错误映射的最小改动，不做未经证明的扩展设计。

## Required Workflow

### 0. 请求分流与交付模式

先判断用户真正要的交付物，避免把轻量请求升级成完整验收报告：

| 模式 | 进入条件 | 交付物 |
|---|---|---|
| `完整对齐验证` | 已有需求/方案/实现/测试用例，用户要判断是否对齐或能否发布 | 完整对齐报告 |
| `失败定位` | 用户只给一个失败用例、缺陷单或异常响应 | 单用例证据链 + 分类结论 + 下一步验证 |
| `测试覆盖评审` | 测试都通过，但用户问覆盖是否足够或能否发版 | 覆盖矩阵 + 缺口清单 + 发布建议 |
| `测试用例校准` | 测试预期和需求/方案冲突，或新增了未确认口径 | 用例口径差异说明 + 待确认问题 |
| `只运行测试` | 用户只要执行某条测试命令，不要求需求/方案/实现对齐分析 | 运行命令并报告结果，不输出完整模板 |

分流后再进入后续步骤。只有 `完整对齐验证` 默认使用完整输出模板；其他模式输出要收敛，避免让用户在一次失败定位里读一整份报告。

### 1. 收集输入并标注权威级别

先列出可用材料，不要直接跳到代码：

| 材料 | 例子 | 角色 |
|---|---|---|
| 需求依据 | PRD、澄清文档、评审结论 | 判断目标口径 |
| 技术方案 | 后端方案、接口方案、流程图 | 判断设计承诺 |
| 实现证据 | commit、diff、类、方法、SQL、配置 | 判断真实落地 |
| 测试证据 | 测试用例、缺陷单、截图、接口响应、日志 | 判断验收结果 |
| 运行证据 | 单测、集成测试、接口调用、SQL 结果 | 判断问题可复现性 |

如果缺少需求或测试用例，先说明当前只能做保守核对；不要把不完整输入包装成最终验收结论。

同时检查测试用例自身是否可执行、可追溯、可复现：

| 检查项 | 需要确认什么 | 常见问题 |
|---|---|---|
| 前置条件 | 数据、权限、配置、缓存、时间窗口、租户/机构范围 | 用例没写前置数据却要求固定结果 |
| 操作步骤 | 请求入口、参数、触发方式、重试/异步等待方式 | 步骤缺失导致无法复现 |
| 预期结果 | 响应字段、状态、统计口径、排序、分页、事件副作用 | 只写“正确展示”，没有可断言结果 |
| 需求来源 | PRD、澄清结论、评审纪要、接口契约 | 测试新增口径但未确认 |
| 环境约束 | 测试环境、依赖服务、mock、开关、版本分支 | 环境与实现分支不一致 |

测试用例缺少前置条件时，先补“用例可执行性问题”，不要直接判实现错误。

### 2. 建立用例到需求的映射

把测试用例拆成可核验条目：

| 用例 ID | 测试步骤 | 预期结果 | 对应需求点 | 对应方案点 | 是否可追溯 |
|---|---|---|---|---|---|
| TC-001 | ... | ... | ... | ... | 是/否 |

判断规则：

- 找不到需求依据的用例，不要立刻判实现错；标记为“测试用例新增口径/待确认”。
- 找得到需求但方案未覆盖，优先判为“方案遗漏”，再判断实现是否也遗漏。
- 需求和方案都覆盖，但实现结果不一致，进入代码调用链追踪。
- 需求、方案、实现一致，但测试预期不一致，判为“测试用例口径偏差”并给出证据。

### 3. 追踪实现真实执行链

对每个失败或高风险用例，按真实路径追踪：

1. 入口：Controller、接口、定时任务、MQ 消费、前端路由或批处理入口
2. 调用链：Service、Manager、Facade、Repository、Mapper、Client
3. 最终数据源/计算点：SQL、缓存、RPC、文件、规则计算、聚合逻辑
4. 实际影响结果的参数：请求字段、实体 ID、状态、时间范围、分页、权限、租户、上下文
5. 守卫条件 vs 真实业务依赖：哪些只是非空/权限/兼容检查，哪些真正参与查询或计算

不要把字段存在、参数透传或非空校验当成真实业务依赖。只有最终 fetch、calculation、assignment 或 persistence 使用了它，才算影响结果。

### 4. 分类每个差异

每个差异只能先归入一个主类，避免一句“可能都有问题”糊过去：

| 分类 | 判断标准 | 下一步 |
|---|---|---|
| 需求缺失/歧义 | 测试预期找不到明确需求依据，或需求之间冲突 | 输出待确认问题 |
| 方案遗漏 | 需求明确，但方案没有覆盖或设计口径错误 | 补方案或评审变更 |
| 实现偏差 | 需求和方案明确，代码路径未按方案执行 | 最小代码修复 |
| 测试用例偏差 | 需求、方案、实现一致，但测试预期不一致 | 建议测试用例修正 |
| 数据/环境问题 | 逻辑正确，但测试数据、配置、权限、缓存或环境状态不满足前提 | 修复环境或补充前置条件 |
| 覆盖不足 | 当前测试未覆盖关键需求、边界或风险 | 补测试用例 |

### 5. 形成修复或对齐建议

按分类给出动作，不要混成一个“大修方案”：

- `实现偏差`：给出最小代码改动点、为什么足够、需要补的自动化测试。
- `方案遗漏`：给出方案补充点、影响范围、是否需要重新评审。
- `需求缺失/歧义`：只提最关键的单点确认问题，写清当前观察、影响点、可选方案、推荐方案、待确认问题。
- `测试用例偏差`：给出应调整的用例预期、步骤或前置数据，并说明需求证据。
- `覆盖不足`：补充缺失用例清单，优先覆盖主链路、边界、权限、状态流转、统计口径、异步链路、批量性能和回归风险。

### 6. 给出发布判断

发布建议必须来自差异分类和验证证据，不要只根据“测试是否通过”下结论：

| 建议 | 条件 |
|---|---|
| `建议发布` | 需求/方案/实现/测试用例已闭环，关键风险有覆盖，剩余问题不影响当前发布目标 |
| `有条件发布` | 存在低风险覆盖缺口或非阻塞用例问题，已列出补测/监控/回滚措施 |
| `暂缓发布` | 存在实现偏差、方案遗漏、高影响需求歧义、关键用例不可复现或环境数据不可信 |
| `无法判断` | 缺少需求、方案、实现证据或测试材料，无法形成闭环证据链 |

如果给出 `有条件发布` 或 `暂缓发布`，必须写清阻塞项、Owner 建议、下一步验证和重新判断条件。

## Common Mistakes

| 错误做法 | 为什么错 | 正确做法 |
|---|---|---|
| 看到测试失败就加 fallback | 可能掩盖需求、方案、数据源不一致 | 先证明真实业务依赖和最终数据源 |
| 把测试用例当成唯一真相 | 测试用例也可能理解错或漏前置条件 | 回到需求和评审结论核对 |
| 只看入口参数和 null check | guard 不代表参与业务计算 | 追到最终查询、计算、赋值点 |
| 只跑测试不看断言含义 | 只能知道红绿，不能知道是否对齐需求 | 建立用例到需求/方案映射 |
| 用单条查询修补批量问题 | 容易引入 N+1 或性能回退 | 收集批量参数后批量查询并映射 |
| 把环境问题改成代码兼容 | 污染业务逻辑，后续更难定位 | 明确测试前置数据、配置和环境状态 |
| 测试都通过就建议发布 | 覆盖不足、用例口径偏差或环境不可信仍可能漏风险 | 先检查覆盖矩阵、关键风险和发布判断条件 |

## Final Check

交付前确认：

- 已列出输入材料和权威级别。
- 已按用户意图选择正确交付模式，没有把单用例定位扩成完整报告。
- 已检查测试用例前置条件、步骤、预期和环境约束。
- 每个失败用例都能追溯到需求、方案、实现和测试证据。
- 已区分 guard 条件和真实业务依赖。
- 已定位最终数据源/计算点/赋值点。
- 已说明关键参数为什么影响结果。
- 未引入未经证明的 fallback 或平行数据源。
- 批量场景没有提出 N+1 查询或循环远程调用。
- 结论明确区分需求问题、方案问题、实现问题、测试用例问题、环境数据问题和覆盖不足。
- 发布建议有明确依据，且阻塞项和重新判断条件清楚。
- 若需要改代码，已给出最小闭环修复和验证方式。
